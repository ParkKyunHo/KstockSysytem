## §0. 시스템 정체성 및 헌법

### 0.1 시스템 한 문장 요약

> **사용자가 정의한 박스 구간을, 시스템이 인내심 있게 지키다가 정확히 포착한다.**

### 0.2 역할 분담

```
사용자 (Human Trader, 8년 전업 경력):
  - 종목 선별 (HTS, 시그널리포트 등 외부 도구)
  - 박스 구간 설정 (가격 범위, 비중, 손절폭)
  - 추가 매수 결정 (불타기)
  - 리포트 요청 시점 결정
  - 시스템 설정 변경

시스템 (V7.1):
  - 추적 (시세 모니터링)
  - 진입 감지 (박스 + 양봉 조건)
  - 자동 매수 (지정 비중)
  - 자동 청산 (손절/익절/TS)
  - 알림 (텔레그램, 웹)
  - 데이터 보존 (DB, 리포트)
  - 리포트 생성 (Opus 4.7, On-Demand)
```

### 0.3 헌법 5원칙

모든 코드와 결정의 절대 기준입니다.

#### 원칙 1: 사용자 판단 불가침

```
사용자가 정의한 박스 구간을 시스템이 임의 수정 금지.

허용:
  - B 수준 경고 (명백한 입력 오류)
  - 예: 박스 상단 < 하단 → "입력 오류" 경고

금지:
  - 박스 자동 조정
  - "이 박스는 위험해 보임" 같은 가치 판단
  - 사용자 비중 자동 축소
  - 손절폭 자동 변경
```

#### 원칙 2: NFR1 최우선 (박스 진입 절대 놓치지 않음)

```
박스 진입 순간 누락 = 시스템 실패.

목표:
  - 시세 모니터링 지연 < 1초 (박스 근접 시)
  - 봉 완성 후 매수 주문까지 < 5초

트레이드오프:
  - 안정성, 부하, 비용 < NFR1
  - "비용 절감" 위해 폴링 간격 늘리는 것 금지
  - "코드 단순화" 위해 모니터링 줄이는 것 금지
```

#### 원칙 3: 충돌 금지

```
V7.0 인프라 유지하면서 V7.1 신규 추가.

규칙:
  - 명명: V71 접두사 또는 src/core/v71/ 패키지
  - 데이터: 기존 테이블 확장만, 파괴적 변경 금지
  - 삭제: 의존성 추적 후 안전한 순서
  - Feature Flag: V7.0 운영 중 V7.1 점진 활성화

예시:
  허용: V71BoxManager, src.core.v71.box.box_manager
  금지: PositionManager 직접 수정 (V7.0과 충돌)
```

#### 원칙 4: 시스템 계속 운영

```
어떤 상황에서도 시스템 가동 유지.

금지:
  - 긴급정지 코드
  - 자동 시스템 종료
  - 무한 재시도 차단

처리:
  - WebSocket 끊김 → 재연결 우선
  - 시스템 재시작 → 자동 복구
  - 빈도 모니터링 → 알림만, 정지 없음
```

#### 원칙 5: 단순함 우선

```
복잡한 예외 처리보다 단순 룰.

원칙:
  - "있으면 좋은 것"과 "필수" 구분
  - 매 결정 시 "정말 필요한가?" 자문
  - 엣지 케이스 일부 감수 < 복잡 룰

예시:
  허용: VI 포함 봉 데이터 그대로 판정
  금지: VI 봉만 특별 처리하는 복잡한 분기
```

### 0.4 NFR (비기능 요구사항)

```
NFR1 (최우선): 박스 진입 절대 놓치지 않음
  - 시세 지연 < 1초 (박스 근접)
  - 매수 주문 지연 < 5초

NFR2: 사용자 판단 존중
  - B 수준 경고만 (입력 오류)
  - 가치 판단 경고 금지

NFR3: 수동-시스템 분리 명확
  - MANUAL 포지션은 시스템 자동 청산 안 함
  - SYSTEM 포지션은 시스템이 관리
  - 혼합 시 명확한 룰 적용
```

### 0.5 본 문서의 적용 범위

```
적용:
  - V7.1 모든 거래 로직
  - 박스 시스템 전체
  - 손절/익절/TS 전체
  - 수동 거래 처리
  - 모든 엣지 케이스

미적용 (별도 문서):
  - 데이터 모델 → 03_DATA_MODEL.md
  - 아키텍처 → 04_ARCHITECTURE.md
  - 보안 → 12_SECURITY.md
  - UI → 10_UI_GUIDE.md
```

## §1. 이중 경로 (Dual Path)

> ⚠️ **PRD Patch #3 (2026-04-25)**: 경로(path_type)는 **박스의 속성**입니다.
> tracked_stocks가 아니라 support_boxes에 저장됩니다.
> 같은 종목 안에서 박스마다 다른 경로 가능합니다.
> 자세한 내용: 13_APPENDIX.md §6.2.Y

V7.1은 두 가지 경로로 박스를 추적합니다.

### 1.1 경로 A: 주도주 단타

```yaml
이름: PATH_A
한국어 이름: 주도주 단타
주 용도: 단기 모멘텀 매매

출처:
  사용자가 외부 도구(HTS, 시그널리포트 등)로 종목 선별
  웹 대시보드에 등록

시간축:
  3분봉 (Three-minute candle)

진입 트리거:
  눌림 (Pullback):
    - 3분봉 N-1: Close > Open AND 박스 내 종가
    - 3분봉 N: Close > Open AND 박스 내 종가
    - 매수 실행: 3분봉 N 완성 직후
  
  돌파 (Breakout):
    - 3분봉 N: 종가 > 박스 상단 AND Close > Open
    - 매수 실행: 3분봉 N 완성 직후

추적 기간:
  무제한 (월 1회 리뷰 알림)

특징:
  - 빠른 시간프레임 (3분)
  - 당일 또는 단기 추세 대응
  - 사용자가 매일 저녁 종목 선별 후 등록
```

### 1.2 경로 B: 수동 중기

```yaml
이름: PATH_B
한국어 이름: 수동 중기 (사용자 직접 분석 종목)
주 용도: 중장기 눌림/돌파

출처:
  사용자 직접 등록 (외부 분석 결과 기반)
  웹 대시보드에 등록

시간축:
  일봉 (Daily candle)

진입 트리거:
  눌림 (Pullback):
    - 일봉 N: Close > Open AND 박스 내 종가
    - 매수 실행: 익일 09:01
  
  돌파 (Breakout):
    - 일봉 N: 종가 > 박스 상단 AND Close > Open
    - 매수 실행: 익일 09:01

매수 시점:
  익일 09:01 (시초가 동시호가 직후)
  09:00 동시호가 참여 안 함

갭업 처리:
  익일 시가 갭업 5% 미만: 정상 매수
  익일 시가 갭업 5% 이상: 매수 포기 + 텔레그램 알림

추적 기간:
  무제한 (월 1회 리뷰 알림)

특징:
  - 일봉 기준
  - 6개월~1년 추적 가능
  - 리포트 없이도 등록 가능
  - 사용자가 깊이 분석한 종목
```

### 1.3 동일 종목 이중 경로

```yaml
허용 여부: 가능 (PRD Patch #3 적용)

구조:
  같은 종목 안에서 PATH_A 박스와 PATH_B 박스 동시 보유 가능
  tracked_stocks 레코드는 1개 (종목당)
  support_boxes 레코드 여러 개, 각 박스가 path_type 보유

예시:
  삼성전자 (tracked_stocks #1):
    박스 1: PATH_A (3분봉), 73,000~74,000원, 비중 5%
    박스 2: PATH_A (3분봉), 70,000~71,000원, 비중 5%
    박스 3: PATH_B (일봉), 65,000~66,000원, 비중 10%
  
  서로 다른 가격대, 서로 다른 경로
  같은 종목이지만 진입 조건은 박스마다 독립

관리:
  각 박스 독립 진입 조건 (path_type별 봉 단위 차이)
  각 박스 독립 손절/익절
  포지션은 박스에서 트리거되어 별도 관리
  
포지션 한도:
  종목당 30% (모든 박스의 활성 포지션 합계)
  경로 구분 없이 통합 관리
  ※ 매수 대기 박스의 비중은 한도에서 제외

주문 태깅:
  키움 계좌는 통합되므로 주문 시 position_id 태그 필수
  (구현 시 키움 API의 client_order_id 지원 여부 확인)
```

### 1.4 경로 간 데이터 분리

```yaml
원칙: 박스는 path_type별로 독립적으로 진입 조건 판정

구조 (PRD Patch #3):
  - tracked_stocks 레코드: 종목당 1개 (path_type 없음)
  - support_boxes 레코드: 박스 수만큼, 각각 path_type 보유
  - positions 레코드: 박스에서 트리거된 매수마다 1개
  - trade_events: 각 포지션별 이벤트

시세 데이터 (종목 단위 공유):
  - 키움 API 시세: 종목당 1회 구독
  - 봉 데이터: PATH_A는 3분봉, PATH_B는 일봉
    → 같은 종목에 두 경로 박스 있으면 둘 다 구독
  - 봉 완성 시 path_type별 박스에 진입 조건 평가

포지션 관리:
  포지션은 box.path_type을 상속
  손절/익절/TS 룰은 path_type별로 차이 없음 (동일)
  → 진입 조건만 path_type별로 다름
  → 매수 후 관리는 동일 룰
```

## §2. 상태 머신 (State Machine)

각 추적 종목(tracked_stock)은 5개 상태 중 하나에 있습니다.

### 2.1 상태 정의

```yaml
TRACKING:
  의미: 추적 등록됐으나 박스 미설정
  사용자: 종목만 등록한 상태
  시스템: 일 1회 종가만 업데이트
  시세 구독: 없음
  특징: 박스 없으니 매수 트리거 없음

BOX_SET:
  의미: 박스 1개 이상 설정됨, 진입 대기
  사용자: 박스 가격, 비중 등 입력 완료
  시스템: 박스 근접도에 따라 시세 모니터링
  시세 구독: 박스 ±5% → 실시간, 그 외 차등
  특징: 진입 트리거 대기 상태

POSITION_OPEN:
  의미: 1개 이상 박스에서 매수 실행됨, 부분 익절 미발생
  사용자: 매수 알림 받음
  시스템: 손절/익절/TS 모니터링
  시세 구독: 상시 WebSocket
  특징: 활성 포지션, 청산 대기

POSITION_PARTIAL:
  의미: 부분 익절 발생 (+5% 또는 +10%)
  사용자: 부분 익절 알림 받음
  시스템: 남은 수량에 대해 청산 관리 계속
  시세 구독: 상시 WebSocket
  특징: 일부 청산됨, 나머지 관리 중

EXITED:
  의미: 전량 청산 완료
  사용자: 청산 알림 받음
  시스템: 모니터링 중단
  시세 구독: 없음
  특징: 자동 재진입 안 함, 수동 재등록 필요
```

### 2.2 상태 전이 규칙

```yaml
TRACKING → BOX_SET:
  조건: 박스 1개 이상 설정
  트리거: 사용자가 웹 대시보드에서 박스 추가
  자동 발생: 즉시

BOX_SET → POSITION_OPEN:
  조건: 박스 진입 + 양봉 조건 충족 + 매수 성공
  트리거: 시스템이 진입 트리거 감지 + 주문 체결
  자동 발생: 즉시

POSITION_OPEN → POSITION_PARTIAL:
  조건: +5% 또는 +10% 부분 익절 실행
  트리거: 시스템이 익절 조건 감지 + 매도 체결
  자동 발생: 즉시

POSITION_PARTIAL → POSITION_PARTIAL:
  조건: 추가 부분 익절 (+5% 후 +10% 도달)
  트리거: 시스템이 두 번째 익절 조건 감지
  자동 발생: 즉시
  (상태 자체는 그대로, 이벤트만 진행)

POSITION_OPEN → EXITED:
  조건: 손절 또는 TS 청산 (전량)
  트리거: 시스템이 청산 조건 감지

POSITION_PARTIAL → EXITED:
  조건: 잔여 수량 손절 또는 TS 청산
  트리거: 시스템이 청산 조건 감지

EXITED → (종료):
  자동 재진입 없음
  사용자가 수동으로 재등록해야 함
  → 새 tracked_stocks 레코드 생성 (TRACKING 상태)
```

### 2.3 자동 이탈 (시스템 자동 EXITED)

```yaml
조건 1: 박스 하단 -20% 이상 이탈
  계산: (현재가 - 박스 최저 하단) / 박스 최저 하단 <= -0.20
  의미: 박스 가정 무너짐

조건 2: 거래 정지
  키움 API 거래 정지 상태 감지
  의미: 거래 불가능

조건 3: 상장 폐지 위험
  관리종목 지정, 투자 경고/위험 종목 지정
  키움 API 상태 코드로 감지

처리:
  자동 EXITED 처리
  텔레그램 알림 (CRITICAL)
  웹 알림 아이콘 활성화
  포지션 보유 시: 시장가 매도 시도 (체결 보장 안 됨)

기록:
  trade_events에 AUTO_EXIT 이벤트
  사유 명시 (BOX_DROP_20 / TRADING_HALTED / DELISTING_RISK)
```

### 2.4 상태별 시세 모니터링

```yaml
TRACKING:
  주기: 일 1회 종가
  방식: REST API (장 마감 후)
  시세 구독: 없음
  부하: 매우 낮음

BOX_SET:
  현재가가 박스 상단 +10% 이내:
    실시간 WebSocket
    부하: 높음
  
  현재가가 박스 상단 +10% ~ +30%:
    3분봉 폴링 (3분마다 1회)
    부하: 중간
  
  현재가가 박스 상단 +30% 이상:
    일 1회 종가 업데이트
    부하: 낮음
  
  박스 상단 ±5% 이내:
    집중 모니터링 모드
    NFR1 보장 (절대 놓치지 않음)

POSITION_OPEN / POSITION_PARTIAL:
  주기: 매 틱 (실시간)
  방식: WebSocket 상시 구독
  목적: 손절/익절/TS 즉시 판정
  부하: 높음 (필수)

EXITED:
  시세 구독 없음
  로그만 보존
```

### 2.5 안전장치: 박스 진입 누락 감지

NFR1 보장을 위한 사후 검증:

```yaml
실행 시점: 매일 장 마감 후 (15:30)

감사 로직:
  for each tracked_stock in BOX_SET:
    당일 모든 봉 데이터 조회
    각 봉마다 박스 진입 조건 체크
    조건 충족했는데 매수 안 된 경우:
      → 알림 (CRITICAL)
      → 로그 기록
      → 사유 분석 (시세 끊김? VI? 시스템 오류?)

조치:
  사후 감사이므로 매수는 안 함
  (이미 시점 지났음)
  사용자에게 보고
  시스템 개선에 활용
```

## §3. 박스 설정 (Box Configuration)

박스는 V7.1의 **핵심 입력**입니다. 사용자가 설정하고 시스템이 지킵니다.

### 3.1 박스 정의

```yaml
박스 (Box, Support Box):
  하나의 매수 계획
  가격 범위 + 비중 + 손절폭으로 정의

데이터 모델:
  - id: UUID
  - tracked_stock_id: 추적 종목 FK
  - box_tier: 박스 순서 (1, 2, 3, ...)
  - upper_price: 박스 상단 (원 단위)
  - lower_price: 박스 하단 (원 단위)
  - position_size_pct: 투입 비중 (%, 총 자본 기준)
  - stop_loss_pct: 손절폭 (기본 -5%, 사용자 변경 가능)
  - strategy_type: 'PULLBACK' | 'BREAKOUT'
  - entry_trigger: 자동 결정 (경로 + 전략 조합)
  - status: 'WAITING' | 'TRIGGERED' | 'INVALIDATED' | 'CANCELLED'
  - memo: 사용자 메모 (선택)
  - created_at: 생성 시각
  - modified_at: 수정 시각
  - expires_at: 만료 시각 (자동 만료 없음, 30일 확인 알림)
```

### 3.2 박스 입력 UI 흐름

### 3.2.1 경로가 박스의 속성인 이유 (설계 근거)

```yaml
이전 (PRD 초기):
  tracked_stocks.path_type 필수
  같은 종목 이중 경로 시 tracked_stocks 2개 레코드
  사용자: 같은 종목이 추적 목록에 2번 표시 → UX 어색

현재 (PRD Patch #3):
  support_boxes.path_type 필수
  tracked_stocks.path_type 제거
  종목당 1개 레코드, 박스마다 path_type
  사용자: 추적 목록에 한 종목은 한 번만 표시 → UX 자연스러움

근거:
  경로는 "종목 자체의 속성"이 아니라
  "이 박스에서 진입을 어떻게 판정할지"의 문제
  즉, 박스의 진입 조건 속성인 것이 자연스러움
```

### 3.3 박스 자동 계산 표시

UI에서 사용자가 입력하면 자동 표시:

```yaml
매수 가능 수량:
  계산: floor((총 자본 × 박스 비중%) / 매수가)
  매수가: 박스 중간값 또는 상단

손절 시 최대 손실액:
  계산: 매수 가능 수량 × 매수가 × 손절폭

손익비 추정:
  +5% 도달 시 수익 / 손절 시 손실
  계산: (매수가 × 0.05 × 30% 수량) / (매수가 × |손절폭| × 100% 수량)
  표시: "1:M.M" 형식

종목당 총 비중:
  현재 박스의 비중 + 같은 종목의 다른 박스 비중 합계
  30% 초과 시 경고
```

### 3.4 박스 제약 조건

```yaml
한 종목 최대 비중:
  30% (실제 포지션 기준)
  ※ 매수 대기 박스의 비중 합계는 한도에서 제외
  
  계산:
    actual_invested = sum(position.weighted_avg_price × position.total_quantity)
    pct = (actual_invested / total_capital) × 100
    pct <= 30이면 새 매수 가능

박스 수:
  무제한 (다층 박스)
  사용자가 원하는 만큼 추가

박스 겹침:
  비허용
  검증:
    box_a.upper > box_b.lower AND box_a.lower < box_b.upper → 겹침
  같은 종목+경로의 박스끼리만 검증
  (다른 종목 또는 다른 경로는 무관)

박스 상단 vs 하단:
  상단 > 하단 필수
  같으면 입력 오류 (B 수준 경고)

박스 진입 순서:
  자유
  1차 박스 미진입 상태에서 2차 박스 진입 가능
  (시장이 1차를 건너뛰고 2차로 갔을 때)
```

### 3.5 한도 계산 상세

```yaml
원칙: 실제 포지션만 한도 계산에 포함

정의:
  position_size_pct = (weighted_avg_price × total_quantity / total_capital) × 100

매수 대기 박스:
  한도에서 제외
  예시:
    총 자본: 1억원
    박스 1: 비중 10% (대기)
    박스 2: 비중 10% (대기)
    박스 3: 비중 10% (대기)
    
    한도 사용: 0% (실제 매수 없음)
    합계 30%인 박스들이 있어도 한도는 0%

매수 실행 후:
  실제 투입 금액으로 계산
  예시:
    박스 1 진입 (1,000만원 투입)
    한도 사용: 10%
    
    박스 2 진입 (1,000만원 투입)
    한도 사용: 20%
    
    박스 3 진입 (1,000만원 투입)
    한도 사용: 30% (한계 도달)
    
    박스 4 진입 시도 (10% 추가)
    → 30 + 10 = 40% > 30% (한도 초과)
    → 매수 거부 + 알림

한도 초과 시 처리:
  매수 시도 자체를 거부
  텔레그램 알림 (HIGH)
  사용자에게 박스 비중 재조정 요청

부분 익절 후 한도:
  실제 보유 수량 기준 재계산
  예시:
    박스 1 진입 (10% 사용, 100주)
    +5% 청산 (30주 매도, 70주 남음)
    한도 사용: 7% (10% × 70%)
    여유 한도: 23% (30% - 7%)

평단가 변경의 한도 영향:
  추가 매수로 평단가 변경되어도
  한도는 "실제 투입 자본" 기준이지 평단가 무관
  계산: weighted_avg_price × total_quantity = 실제 투입액
```

### 3.6 박스 수정 정책

```yaml
BOX_SET 상태 (미진입):
  허용:
    - 박스 가격 수정 (상단/하단)
    - 비중 수정
    - 손절폭 수정
    - 박스 삭제
    - 박스 추가
  제한 없음

POSITION_OPEN / POSITION_PARTIAL 상태:
  이미 매수한 박스:
    수정 불가 (매수가 확정됨)
    삭제 불가
  
  미진입 박스:
    가격 수정: 가능
    비중 수정: 가능
    손절폭 수정: 가능 (단, 완화 방향 시 경고)
    박스 삭제: 가능
    새 박스 추가: 가능

손절폭 완화 시 경고:
  완화: -5% → -7% 같이 더 큰 손실 허용 방향
  타이트: -5% → -3% 같이 더 작은 손실 방향
  
  완화 방향 수정 시:
    UI에 경고 표시
    "리스크가 증가합니다. 정말 진행하시겠습니까?"
    사용자 확인 후 적용

모든 박스 수정:
  수정 로그 필수 (audit trail)
  modified_at 갱신
  변경 사항 기록 (이전값 → 새 값)
  사용자가 입력한 사유 (선택)
```

### 3.7 박스 자동 만료 정책

```yaml
원칙: 자동 삭제 없음

확인 알림:

  - 유지: 아무 행동 안 함 (다음 30일 후 다시 알림)

  - 삭제: 박스 명시적 삭제

  60일+ 정체된 박스 별도 강조
```

### 3.8 박스 진입 조건 (눌림, 경로 A)

```yaml
타임프레임: 3분봉

조건 (모두 충족):
  조건 1 (직전 3분봉, N-1):
    Close > Open  (양봉)
    Close >= 박스 하단
    Close <= 박스 상단
  
  조건 2 (현재 3분봉, N):
    Close > Open  (양봉)
    Close >= 박스 하단
    Close <= 박스 상단

매수 실행 시점:
  3분봉 N 완성 직후 (즉시)

VI 처리:
  VI 포함 봉도 정상 판정
  Close > Open이면 양봉으로 인정
  (VI 중 체결 적어도 시가/종가는 형성됨)

엣지 케이스:
  Close == Open: 양봉 아님 (조건 미충족)
  거래 없는 봉: 시가=종가=고가=저가, 양봉 아님
```

### 3.9 박스 진입 조건 (돌파, 경로 A)

```yaml
타임프레임: 3분봉

조건 (모두 충족):
  조건 1 (현재 3분봉, N):
    Close > 박스 상단  (돌파)
    Close > Open  (양봉)

매수 실행 시점:
  3분봉 N 완성 직후 (즉시)

박스 하단 의미:
  유효 돌파 범위 필터
  (당신 결정: 갭 차이 큰 것 허용, 즉 박스 하단은 참고만)
  
  실제 적용:
    봉의 시가가 박스 하단 이상이어야 함
    (봉 자체가 박스 위로 갭업한 경우 제외)
  
  검증:
    Open >= 박스 하단

상한가 임박:
  당신 결정: 시스템 룰 그대로 매수 (옵션 A)
  근거: 상따 매매도 명확한 손절 룰 있음

VI 처리:
  단일가 매매 참여 (지정가)
  실패 시 시장가
  갭 3% 이상 시 포기 (§10.3 참조)
```

### 3.10 박스 진입 조건 (눌림, 경로 B)

```yaml
타임프레임: 일봉

조건 (모두 충족):
  조건 1 (당일 일봉, N):
    Close > Open  (양봉)
    Close >= 박스 하단
    Close <= 박스 상단

매수 실행 시점 (1차):
  익일 09:01

매수 방식 (1차):
  지정가 1호가 위 → 5초 × 3회 → 시장가 (§4 참조)

매수 실행 시점 (2차, 안전장치):
  익일 09:05 (1차가 실패한 경우만)
  
  1차 실패 정의 (안전장치 발동 조건):
    - 시초 동시호가 결과 시초가 VI 발동 → 단일가 매매 진입 → 미체결
    - 09:01 시점 VI 발동 중이라 단일가 매매에서 미체결
    - 키움 API 일시 장애로 09:01 주문 실패
    - 그 외 1차 매수 시도가 09:04:59까지 미체결로 종료된 경우
  
  안전장치 적용 안 함 (포기 그대로):
    - 갭업 5% 초과로 매수 포기 (§3.10 갭업 룰)
    - 추적 종료 (EXITED) 상태
    - 거래 정지 / 상장폐지 위험 종목
    - 한도 초과 (30% 한도 도달)
  
  매수 방식 (2차):
    시장가 (즉시 체결 우선)
    근거: 1차 미체결 = 가격 변동 가능성 → 추가 지연 회피
  
  2차 시점 갭업 재검증:
    측정: (현재가 - 전일 종가) / 전일 종가
    < 5%: 시장가 매수 진행
    >= 5%: 매수 포기 + 알림 (안전장치도 무력화)
  
  알림 (2차):
    매수 성공: HIGH "[종목명] 09:05 안전장치 매수 성공 (1차 실패 사유: VI/미체결/...)"
    매수 포기 (재갭업): HIGH "[종목명] 09:05 안전장치 시점 갭업 X.X%, 매수 포기"

갭업 5% 이상 시 (1차 시점):
  매수 포기 (안전장치 미발동)
  텔레그램 알림 (HIGH)
  내용: "[종목명] 갭업 5% 이상, 매수 포기"
  박스는 유지 (다음 기회)

판정 시점:
  당일 15:30 장 마감 후
  일봉 종가 확정
  조건 충족 시 익일 매수 예약
```

### 3.11 박스 진입 조건 (돌파, 경로 B)

```yaml
타임프레임: 일봉

조건 (모두 충족):
  조건 1 (당일 일봉, N):
    Close > 박스 상단  (돌파)
    Close > Open  (양봉)
    Open >= 박스 하단  (갭업 아닌 정상 돌파)

매수 실행 시점 (1차):
  익일 09:01

매수 실행 시점 (2차, 안전장치):
  익일 09:05 (§3.10과 동일 안전장치)
  
  1차 실패 시 09:05 시장가 매수 재시도:
    - 시초가 VI / 단일가 미체결 / API 장애 등
    - 갭업 5% 초과 (1차 시점)는 안전장치 미발동
    - 09:05 시점 갭업 재검증 < 5% 시 시장가 매수
  
  상세: §3.10 동일 룰 적용

갭업 5% 이상 시 (1차 시점):
  매수 포기 (눌림과 동일, 안전장치 미발동)
```

### 3.12 동일 종목 다중 전략

```yaml
허용:
  같은 종목에 눌림 + 돌파 동시 설정 가능
  같은 종목에 다층 박스 (1차, 2차, 3차) 가능
  같은 종목에 경로 A + 경로 B 동시 가능

분리 관리:
  각 박스 독립 레코드
  각 박스 독립 진입 감시
  
박스 겹침 검증:
  같은 종목 + 같은 경로의 박스끼리만 겹침 검증
  다른 경로 또는 다른 종목은 무관

예시:
  에프알텍:
    PATH_A 박스 1 (눌림): 17,000~17,500 (10%)
    PATH_A 박스 2 (눌림): 15,500~16,000 (10%)
    PATH_A 박스 3 (돌파): 22,000 상단 (15%)
    PATH_B 박스 1 (눌림): 14,000~14,500 (10%)
  
  PATH_A 박스 1과 박스 2: 겹치지 않음 (검증 통과)
  PATH_A 박스 3: 다른 가격대 (검증 통과)
  PATH_A와 PATH_B: 다른 경로 (겹침 검증 안 함)
  
  총 박스 4개, 모두 유효
```

### 3.13 박스 상태 변경

```yaml
박스 status 필드의 라이프사이클:

WAITING (초기 상태):
  박스 생성 시 기본값
  진입 트리거 대기 중

TRIGGERED:
  매수 실행 완료 시
  positions 레코드와 연결됨

INVALIDATED:
  시나리오 C 발생 시 (사용자 수동 매수로 박스 무효화)
  또는 자동 이탈 (-20% 이탈 등)
  레코드 삭제 안 함, 이력 보존

CANCELLED:
  사용자가 박스 삭제
  레코드 삭제 (또는 status만 변경, 구현 시 결정)
```

## §4. 매수 실행 (Buy Execution)

박스 진입 트리거 충족 후 매수 실행 룰입니다.

### 4.1 주문 타입

```yaml
기본: 지정가 (Limit Order)
실패 시: 시장가 (Market Order)

지정가 가격:
  현재 호가의 1호가 위
  
  예시:
    현재 매수 1호가: 18,000원
    현재 매도 1호가: 18,050원
    지정가 매수 가격: 18,050원 (매도 1호가 위는 동시 체결 의미)
    
  ※ 정확한 표현: "매수 시 매도 1호가에 주문"
    빠른 체결 위해 매도 호가에 매수 주문
```

### 4.2 지정가 재시도 사이클

```yaml
시도 1:
  지정가 (매도 1호가) 주문
  대기: 5초
  
  체결 → 성공 (사이클 종료)
  미체결 → 시도 2

시도 2:
  주문 취소 후 재발송
  지정가 (현재 매도 1호가, 가격 변경됨 가능)
  대기: 5초
  
  체결 → 성공
  미체결 → 시도 3

시도 3:
  주문 취소 후 재발송
  지정가 (현재 매도 1호가)
  대기: 5초
  
  체결 → 성공
  미체결 → 시장가 전환

시도 4 (최종):
  주문 취소
  시장가 매수 주문
  
  체결 → 성공 (대부분의 경우)
  미체결 → §12.1 시장가 실패 처리
```

### 4.3 부분 체결 처리

```yaml
시나리오:
  지정가 100주 주문
  50주만 체결, 50주 미체결
  대기 시간 5초 종료

처리:
  미체결 50주에 대해 재시도 사이클 계속
  체결된 50주는 그대로 유지 (이미 매수됨)
  
  시도 카운트:
    50주 체결도 1회 시도로 카운트
    나머지 50주에 대해 시도 2 진행

평단가 처리:
  부분 체결된 50주의 평단가 기록
  후속 체결 시 가중 평균 재계산
  
  예시:
    50주 × 18,000 = 900,000
    50주 × 18,050 = 902,500
    합계 100주, 평단가 18,025
```

### 4.4 대량 주문 처리 (호가 소진)

```yaml
상황:
  매수 수량 1,000주
  매도 1호가에 100주만 있음

처리:
  방식: 1호가씩 연속 매수
  
  Step 1: 매도 1호가 100주 매수 (체결)
  Step 2: 매도 1호가가 다음 호가로 이동
          새 1호가에서 매수 시도
  Step 3: 반복
  
  목표 수량 도달까지 계속

위험:
  여러 호가 소진하며 평단가 상승
  슬리피지 발생

알림:
  대량 주문 시 슬리피지 N% 이상이면 알림
  (구현 시 임계치 결정)
```

### 4.5 매수 시점

```yaml
경로 A:
  조건 충족 봉 (3분봉 N) 완성 직후
  지연 < 5초 (NFR1)

경로 B:
  익일 09:01
  09:00 동시호가는 참여 안 함 (변동성 회피)
  09:01 정확한 시점에 주문 발송

매수 봉 결정:
  조건 충족된 봉의 종가 시점부터 카운트
  
  예시:
    14:30~14:33 3분봉
    14:33 봉 완성 (종가 확정)
    14:33:01 ~ 14:33:05 사이 매수 주문 발송
```

### 4.6 갭업 처리 (경로 B만)

```yaml
경로 B 익일 시가 매수 시:

09:01 시점 갭업률:
  계산: (당일 시가 - 전일 종가) / 전일 종가 × 100

갭업 5% 미만:
  정상 매수 (지정가 → 시장가)

갭업 5% 이상:
  매수 포기
  알림 (HIGH):
    "[종목명] 갭업 X.X%, 매수 포기"
  박스는 유지
  다음 진입 트리거 대기

이유:
  너무 큰 갭업은 추격 매수 불리
  눌림/돌파 의미 변질
  단 박스는 유효 → 향후 가격 조정 시 재진입 가능
```

### 4.7 상한가 임박 매수

```yaml
사용자 결정: 시스템 룰 그대로 매수 (상따 매매 허용)

근거:
  - 상한가 매매도 명확한 전략
  - 손절 룰 자동 적용 (-5%)
  - 시스템 단순성 유지

실행:
  현재가 +28% (상한가 직전)
  박스 진입 트리거 충족
  → 정상 매수 실행

위험 인지:
  손익비 불리 (이익 +2%, 손실 -7% 가능)
  사용자가 박스 설정 시 인지하고 입력함
  시스템은 룰만 충실히 실행
```

### 4.8 매수 실패 처리

```yaml
실패 케이스:
  - 시장가 주문도 미체결 (드물지만 가능)
  - 키움 API 응답 오류
  - 잔고 부족
  - 일중 매수 한도 초과

처리:
  Step 1: 시장가 1회 재시도 (§12.1)
  Step 2: 실패 시 텔레그램 알림 (HIGH)
  Step 3: 자동 재시도 중단
  Step 4: 사용자 수동 개입 대기

알림 내용:
  종목명, 시도 횟수, 실패 사유
  "[종목명] 매수 실패: [사유], 수동 확인 필요"

박스 처리:
  박스는 유지 (다음 진입 트리거 대기)
  단, 잔고 부족 등 영구 사유는 사용자 결정 필요
```

### 4.9 매수 후 처리

```yaml
체결 확인 후:
  Step 1: positions 레코드 생성 또는 업데이트
    - source: SYSTEM_A or SYSTEM_B
    - initial_avg_price: 체결 평단가
    - weighted_avg_price: 동일
    - total_quantity: 체결 수량
    - status: OPEN
    - profit_5_executed: False
    - profit_10_executed: False
    - ts_activated: False
  
  Step 2: support_box.status 변경
    WAITING → TRIGGERED
  
  Step 3: tracked_stocks.status 변경
    BOX_SET → POSITION_OPEN
  
  Step 4: 손절선 계산
    fixed_stop = weighted_avg_price × 0.95
  
  Step 5: 시세 구독 변경
    실시간 WebSocket 상시 구독
  
  Step 6: trade_events 기록
    event_type: BUY_EXECUTED
    price, quantity, timestamp
  
  Step 7: 텔레그램 알림 (HIGH)
    매수 정보 발송

다층 박스의 경우:
  2차 박스 매수 시 평단가 재계산
  이벤트 리셋 (§6 참조)
```

## §5. 매수 후 관리 (Position Management)

V7.1의 가장 복잡한 영역입니다. 손절/익절/TS의 정확한 로직.

### 5.1 손절 (Stop Loss) - 절대 원칙

```yaml
조건:
  (현재가 - 평단가) / 평단가 <= -0.05  (단계 1, 매수 직후)
  
  단계별 변경 (5.4 참조):
    매수 ~ +5% 미만: 평단가 -5%
    +5% 청산 후: 평단가 -2%
    +10% 청산 후: 평단가 +4%

판정 주기:
  매 틱 (WebSocket 체결가 수신할 때마다)
  현재가 기준 (장 중 일시 터치도 감지)

실행:
  시장가 매도 전량
  지연 < 1초

WebSocket 끊김 시:
  REST API 폴링 5초 간격
  재연결 시 즉시 재평가

예외:
  없음 (Trend Hold Filter 폐기)
  블랙스완 손실 감수 (전쟁, 횡령 공시 등)
  
일시 터치 처리:
  -5% 일시 터치 후 즉시 반등 → 청산 (이미 시장가 주문 발송)
  당신 결정: "어쩔 수 없이 손절, 원칙이기 때문에"
```

### 5.2 분할 익절 1차 (+5%)

```yaml
조건:
  (현재가 - 평단가) / 평단가 >= 0.05

청산 수량:
  현재 보유 수량의 30%
  계산: floor(total_quantity × 0.30)
  소수점은 반올림 처리 (큰 경로 우선)

주문 방식:
  Step 1-3: 지정가 (매수 1호가, 즉 매도 시 매수자 호가)
            5초 대기 × 3회
  Step 4: 시장가 전환

실행 후 변경:
  position.profit_5_executed = True
  손절선 상향: 평단가 -2%
  TS 활성화: BasePrice 추적 시작
    (TS 청산선은 아직 비교 미사용, +10% 후부터 유효)

trade_events 기록:
  event_type: PROFIT_TAKE_5
  price, quantity, timestamp

알림:
  텔레그램 HIGH
```

### 5.3 분할 익절 2차 (+10%)

```yaml
조건:
  (현재가 - 평단가) / 평단가 >= 0.10
  AND profit_5_executed == True (1차 먼저 발생)

청산 수량:
  현재 보유 수량의 30% (1차 청산 후 남은 70%의 30%)
  
  예시:
    초기 100주
    +5% 청산: 30주 → 남은 70주
    +10% 청산: 70 × 30% = 21주 → 남은 49주

주문 방식:
  지정가 3회 → 시장가 (1차와 동일)

실행 후 변경:
  position.profit_10_executed = True
  손절선 상향: 평단가 +4%
  TS 청산선 유효화 (이 시점부터 비교 시작)

trade_events 기록:
  event_type: PROFIT_TAKE_10
  price, quantity, timestamp
```

### 5.4 손절선 단계별 상향

```yaml
단계 1 (매수 ~ +5% 미만):
  손절선: 평단가 × 0.95 (= -5%)
  TS: 비활성

단계 2 (+5% 청산 완료):
  손절선: 평단가 × 0.98 (= -2%)
  TS: 내부 추적 시작 (BasePrice 기록)
       청산선 비교에 미사용

단계 3 (+10% 청산 완료):
  손절선: 평단가 × 1.04 (= +4%)
  TS: 청산선 유효화 (비교 시작)

상향만, 하향 안 함:
  단계 후퇴 없음
  한 번 +4% 상향됐으면 -2%로 못 돌아감

평단가 변경 시 (추가 매수):
  이벤트 리셋 (§6 참조)
  단계 1로 복귀 (새 평단가 -5%)
```

### 5.5 Trailing Stop (TS)

```yaml
활성화 시점:
  +5% 도달 시
  내부 BasePrice 추적 시작
  단, 청산선 비교에는 미사용

청산선 유효 시점:
  +10% 청산 완료 후
  이때부터 effective_stop 계산에 참여

BasePrice (기준가):
  매수 후 최고가
  실시간 업데이트
  공식: max(현재까지의 모든 봉의 high)
  3분봉 단위로 추적
  
  ※ V7.0의 "Highest(High, 20)" 방식 폐기
  ※ V7.1은 "매수 후 시작 시점부터의 최고가" 단순 추적

ATR 계산:
  기간: 10
  타임프레임: 3분봉
  방법: Wilder's RMA (V7.0 indicator_library 활용)

ATR 배수 단계:
  +10% ~ +15%: 4.0
  +15% ~ +25%: 3.0
  +25% ~ +40%: 2.5
  +40% 이상:   2.0
  
  단방향 축소만 (배수 작아지기만, 커지지 않음)

TS 청산선:
  공식: BasePrice - ATR(10) × 현재 배수
  
  예시:
    BasePrice: 115,000
    ATR(10): 1,500
    현재 배수: 4.0 (수익 +12% 가정)
    TS 청산선: 115,000 - 6,000 = 109,000

업데이트:
  새 봉 완성마다 재계산
  BasePrice 갱신 가능
  ATR 갱신 가능
  배수는 수익률에 따라 결정
  
  TS 청산선 자체는 단방향 (상승만):
    new_ts = BasePrice - ATR × 배수
    if new_ts > current_ts:
      current_ts = new_ts
    else:
      유지 (낮아지지 않음)
```

### 5.6 유효 청산선 계산

```yaml
공식:
  effective_stop = max(고정 손절선, TS 청산선 if 유효)

매 틱 계산:
  현재가 <= effective_stop 이면 청산

상황별 effective_stop:

  단계 1 (매수 ~ +5% 미만):
    고정 손절선: 평단가 × 0.95
    TS: 비활성
    effective_stop = 고정 손절선

  단계 2 (+5% 청산 후, +10% 미만):
    고정 손절선: 평단가 × 0.98
    TS: 내부 추적만, 청산선 비교 미사용
    effective_stop = 고정 손절선

  단계 3 (+10% 청산 후):
    고정 손절선: 평단가 × 1.04
    TS 청산선: BasePrice - ATR × 배수
    effective_stop = max(고정 손절선, TS 청산선)
    
    예시 1: TS가 더 높음
      고정: 104,000 (평단가 100,000 × 1.04)
      TS: 109,000 (BasePrice 115,000 - 6,000)
      effective: 109,000

    예시 2: 고정이 더 높음 (수익 못 키울 때)
      고정: 104,000
      TS: 99,000 (큰 ATR로 인해)
      effective: 104,000
      → 고정 손절선이 받쳐줌

청산 실행:
  현재가 <= effective_stop
  → 시장가 매도 전량
```

### 5.7 Trend Hold Filter 폐기

```yaml
V7.0의 Trend Hold Filter:
  TrendHold = EMA20 > EMA60 AND HighestHigh(20) >= HighestHigh(60) AND ATR유지
  Exit = NOT TrendHold AND Close < TrailingStop
  
  → V7.1에서 완전 폐기

이유:
  - 분할 익절로 이미 수익 일부 확정
  - 추가 보호 불필요
  - 단순성 우선

V7.1 청산:
  유효 청산선 도달 시 무조건 청산

```

### 5.8 Max Holding 정책

```yaml
  → 폐기

이유:

  강제 청산은 단순 룰에 반함

월 1회 리뷰 알림:
  90일+ 보유 종목 강조
  사용자가 직접 청산 결정
```

### 5.9 청산 후 상태 변경

```yaml
부분 청산 (+5% 또는 +10%):
  positions.total_quantity 감소
  positions.weighted_avg_price 유지
  positions.status: OPEN → PARTIAL_CLOSED 또는 PARTIAL_CLOSED 유지
  tracked_stocks.status: POSITION_OPEN → POSITION_PARTIAL
  
  trade_events 기록:
    PROFIT_TAKE_5 또는 PROFIT_TAKE_10

전량 청산 (손절 또는 TS):
  positions.total_quantity = 0
  positions.status: PARTIAL_CLOSED → CLOSED
  tracked_stocks.status: POSITION_OPEN/PARTIAL → EXITED
  
  미진입 박스 처리:
    같은 tracked_stocks의 미진입 박스
    status: WAITING → CANCELLED
    (자동 재진입 안 하므로)
  
  trade_events 기록:
    STOP_LOSS, TS_EXIT 등
  
  시세 구독 해제

알림:
  HIGH (부분) 또는 CRITICAL (손절)
  텔레그램 + 웹 알림 아이콘 (CRITICAL/HIGH는 둘 다)
```

## §6. 평단가 관리 (Average Price Management)

평단가는 V7.1의 미묘한 영역입니다. **매수와 매도의 처리 방식이 다릅니다.**

### 6.1 핵심 원칙

```yaml
원칙 1: 매수 시 평단가 재계산
  새 자금이 들어오므로 가중 평균 다시 계산

원칙 2: 매도 시 평단가 변경 없음
  남은 수량은 여전히 그 가격에 매수된 것
  매도는 "차익 실현"이지 "매수 이력 변경"이 아님
```

### 6.2 매수 시 평단가 재계산

```yaml
신규 매수 (첫 매수):
  position.weighted_avg_price = buy_price
  position.initial_avg_price = buy_price
  position.total_quantity = buy_qty
  
  이벤트 초기화:
    profit_5_executed = False
    profit_10_executed = False
    ts_activated = False
    ts_base_price = buy_price

추가 매수 (다층 박스, 불타기, 수동 합산):
  공식:
    new_total_qty = existing_qty + new_qty
    new_weighted_avg = (existing_qty × existing_avg + new_qty × new_price) / new_total_qty
  
  position.weighted_avg_price = new_weighted_avg
  position.total_quantity = new_total_qty
  position.initial_avg_price 유지 (변경 없음)
  
  이벤트 리셋:
    profit_5_executed = False  ← 핵심: 새 평단가 기준으로 재판정
    profit_10_executed = False
  
  TS BasePrice:
    유지 (최고가 이력은 보존)
  
  손절선 재계산:
    fixed_stop = new_weighted_avg × 0.95  ← 단계 1로 복귀
  
  trade_events:
    event_type: PYRAMID_BUY (다층) 또는 MANUAL_PYRAMID_BUY (수동)
```

### 6.3 추가 매수 시 이벤트 리셋의 의미

```yaml
시나리오:
  1차 매수: 180,000 × 100주
    평단가 180,000
  
  +5% 도달 (189,000), 30주 청산 실행
    profit_5_executed = True
    남은 70주, 평단가 180,000 유지 (매도는 재계산 안 함)
  
  2차 박스 매수: 175,000 × 100주
    새 평단가 = (70 × 180,000 + 100 × 175,000) / 170
              = 177,058
    
    이벤트 리셋:
      profit_5_executed = False  ← 새 +5% = 185,911 기준 다시 판정
      profit_10_executed = False  ← 새 +10% = 194,764 기준 다시 판정
    
    손절선:
      새 평단가 -5% = 168,205
  
  새 +5% (185,911) 도달 시:
    현재 보유 170주의 30% = 51주 청산
    profit_5_executed = True (다시)
```

### 6.4 매도 시 평단가 유지

```yaml
부분 익절 (+5% 또는 +10%):
  position.weighted_avg_price 변경 없음
  position.total_quantity 감소
  이벤트 이력 유지 (profit_5_executed True 유지)
  
예시:
  매수 100주, 평단가 180,000
  +5% 청산 30주 매도 (189,000)
  → 남은 70주, 평단가 180,000 유지
  → 다음 +10% 판정 기준: 180,000 × 1.10 = 198,000
  
수학적 일관성:
  매수 평단가는 "실제 투입 자본"의 가중 평균
  매도는 그 자본에서 차익 실현
  남은 수량의 매수 비용은 변하지 않음

TS 처리:
  BasePrice는 유지 (최고가 추적 계속)
  ts_activated 상태 유지
```

### 6.5 손절 후 처리

```yaml
손절 발생 (전량 청산):
  position.total_quantity = 0
  position.status = CLOSED
  weighted_avg_price 기록 보존 (이력)
  
  연결된 미진입 박스:
    같은 tracked_stock의 다른 박스 (WAITING 상태)
    → status: CANCELLED (자동 재진입 안 함)
  
  tracked_stock.status:
    → EXITED
  
  사용자가 재추적 원하면:
    수동 재등록 필요 (새 tracked_stocks 레코드)
```

### 6.6 평단가 관련 매직 넘버 금지

```yaml
코드 작성 원칙:
  매직 넘버 직접 사용 금지
  
금지 예시:
  if (current - position.avg) / position.avg <= -0.05:  # 매직 넘버
      sell()
  
  position.avg_price = (qty1 * price1 + qty2 * price2) / total  # 직접 계산

허용 예시 (스킬 사용):
  from src.core.v71.position_skill import update_position_after_buy
  
  position = update_position_after_buy(
      position=position,
      buy_price=buy_price,
      buy_qty=buy_qty
  )
  # 함수 내부에서 가중평균 재계산 + 이벤트 리셋
  
  from src.core.v71.exit_skill import calculate_effective_stop
  
  result = calculate_effective_stop(
      position=position,
      current_price=current_price,
      base_price=ts_base_price,
      atr_value=atr_10
  )
  if result.should_exit:
      execute_market_sell(position)

하네스가 강제:
  Trading Rule Enforcer (하네스 3)
  매직 넘버 사용 시 빌드 차단
```

## §7. 수동 거래 시나리오 (Manual Trade Scenarios)

사용자가 HTS로 직접 거래할 수 있습니다. 시스템이 이를 감지하고 적절히 처리합니다.

### 7.1 감지 메커니즘

```yaml
메커니즘:
  PositionSyncManager가 주기적으로 키움 API 잔고 조회
  시스템 DB의 positions와 비교
  차이 발견 시 시나리오별 처리

주기:
  - 시스템 시작 시
  - 매 5분마다 (정기)
  - 사용자 요청 시 (텔레그램 명령어)
  - 의심스러운 거래 발생 후 (수동 매매 알림 후)

감지 방식:
  키움 잔고: 종목별 보유 수량
  시스템 DB: SYSTEM_A + SYSTEM_B + MANUAL 합계
  
  차이 = 키움 잔고 - 시스템 DB 합계
  
  차이 > 0: 사용자 매수 발생 (DB에 없는 보유)
  차이 < 0: 사용자 매도 발생 (DB에 있는데 키움에 없음)
  차이 = 0: 일치 (정상)
```

### 7.2 시나리오 A: 시스템 포지션 + 사용자 추가 매수

```yaml
상황:
  시스템이 이미 매수한 종목
  사용자가 HTS로 같은 종목 추가 매수
  
예시:
  경로 A 시스템 포지션: 100주 @ 180,000원
  사용자 HTS 매수: 50주 @ 185,000원 (불타기)
  키움 잔고: 150주
  시스템 DB: 100주 (불일치)

처리:
  Step 1: 차이 50주 감지
  Step 2: 해당 경로 포지션에 합산
    new_qty = 100 + 50 = 150
    new_avg = (100 × 180,000 + 50 × 185,000) / 150 = 181,667
  
  Step 3: 평단가 재계산 (§6.2)
  Step 4: 이벤트 리셋
    profit_5_executed = False
    profit_10_executed = False
  
  Step 5: 손절선 재계산
    new_stop = 181,667 × 0.95 = 172,584 (단계 1로 복귀)
  
  Step 6: TS BasePrice 유지
  
  Step 7: trade_events 기록
    event_type: MANUAL_PYRAMID_BUY
    quantity: 50, price: 185,000
  
  Step 8: 텔레그램 알림 (HIGH)
    "수동 추가 매수 감지: [종목명] 50주 @ 185,000원"
    "포지션 합산: 150주, 평단가 181,667원"
    "이벤트 리셋, 새 평단가 기준 재판정"

경로 결정:
  단일 경로 보유 시: 그 경로에 합산
  이중 경로 보유 시: 비례 차감과 같은 방식으로 비례 합산?
  
  ※ 사용자 결정 필요 (현재 미정)
  → 디폴트: 큰 경로 우선 합산 (또는 알림 후 사용자 선택)
```

### 7.3 시나리오 B: 시스템 포지션 + 사용자 부분 매도

```yaml
상황:
  시스템 포지션 보유 중
  사용자가 HTS로 일부 매도

경우 1: 단일 경로 + MANUAL 없음
  처리:
    시스템 포지션 수량 감소
    평단가 유지 (매도는 재계산 안 함)
    이벤트 이력 유지
  
  예시:
    경로 A: 100주
    사용자 매도: 50주
    → 경로 A 포지션 50주
    → 평단가 그대로
    → profit_5_executed 등 이력 유지
  
  trade_events:
    event_type: MANUAL_PARTIAL_EXIT
    quantity: 50
  
  알림:
    텔레그램 HIGH
    "수동 부분 매도 감지: [종목명] 50주"
    "시스템 포지션 50주 남음"

경우 2: 단일 경로 + MANUAL 있음
  처리 우선순위:
    1. MANUAL 포지션 먼저 차감
    2. MANUAL 부족 시 시스템 포지션 차감
  
  예시:
    경로 A: 100주
    MANUAL: 30주
    사용자 매도: 50주
    → MANUAL 30주 모두 소진 (CLOSED)
    → 시스템 경로 A 20주 차감 → 80주
  
  알림:
    각 차감마다 별도 trade_events

경우 3: 이중 경로 + 판단 불가
  상황:
    경로 A: 100주
    경로 B: 100주
    사용자 매도: 50주 (어느 경로?)
  
  처리: 자동 비례 차감
    경로 A 차감: 50 × (100/200) = 25주
    경로 B 차감: 50 × (100/200) = 25주
  
  소수점 처리:
    큰 경로 우선 반올림
    예: 경로 A 100주 + 경로 B 50주 = 150주, 매도 10주
        비례: 6.67 + 3.33
        반올림: 큰 경로 7, 작은 경로 3 (합 10)
  
  알림:
    텔레그램 HIGH
    "이중 경로 수동 매도 감지: 자동 비례 차감 적용"
    "경로 A: 25주 차감 → 75주"
    "경로 B: 25주 차감 → 75주"

경우 4: 이중 경로 + MANUAL 있음
  처리 우선순위:
    1. MANUAL 먼저 소진
    2. 그래도 부족하면 이중 경로 자동 비례 차감
```

### 7.4 시나리오 C: 추적 중 미진입 종목 + 사용자 매수

```yaml
상황:
  종목 추적 중 (BOX_SET 상태)
  박스 진입 트리거 충족 안 됨
  사용자가 HTS로 직접 매수

예시:
  종목: 삼성전자
  박스 1: 73,000~74,000원 (대기 중)
  박스 2: 70,000~71,000원 (대기 중)
  박스 3: 67,000~68,000원 (대기 중)
  현재가: 75,500원 (모든 박스 위)
  
  사용자 HTS 매수: 50주 @ 75,500원

처리:
  Step 1: 추적 종료
    tracked_stocks.status: BOX_SET → EXITED
  
  Step 2: 모든 박스 무효화
    support_boxes.status: WAITING → INVALIDATED
    물리적 삭제 X (이력 보존)
  
  Step 3: MANUAL 포지션 신규 생성
    positions:
      source: MANUAL
      tracked_stock_id: NULL (또는 기존 ID 보존)
      weighted_avg_price: 75,500
      total_quantity: 50
  
  Step 4: 시세 구독 변경
    BOX_SET 모니터링 → 시세 구독 해제
    (MANUAL은 시스템 청산 안 함)
  
  Step 5: 텔레그램 알림 (HIGH)
    텔레그램 메시지 (간소):
      "[수동 매수 감지 - 시스템 추적 종료]"
      "종목: 삼성전자 (005930)"
      "매수: 50주 @ 75,500원"
      "시스템 추적 종료, 박스 무효화"
      "이 수동 포지션은 시스템이 자동 관리하지 않습니다."
  
  Step 6: 웹 알림 아이콘 활성화
    상세 내역 (클릭 시):
      "무효화된 박스: 3개"
      " 1차: 73,000원 ~ 74,000원 (10%)"
      " 2차: 70,000원 ~ 71,000원 (10%)"
      " 3차: 67,000원 ~ 68,000원 (10%)"
      "필요 시 재추적 등록 가능"

사용자가 재추적 원하면:
  웹 대시보드에서 새로 종목 등록
  새 tracked_stocks 레코드 (TRACKING 상태)
  박스 다시 설정
```

### 7.5 시나리오 D: 추적 안 하는 종목 + 사용자 매수

```yaml
상황:
  시스템에 등록되지 않은 종목
  사용자 HTS로 매수

예시:
  사용자가 즉흥적으로 매수
  시스템에는 추적 정보 없음

처리:
  Step 1: MANUAL 포지션 신규 생성
    positions:
      source: MANUAL
      tracked_stock_id: NULL
      weighted_avg_price: 매수가
      total_quantity: 매수 수량
  
  Step 2: tracked_stocks 연결 없음
    별도 추적 등록 안 함
  
  Step 3: 텔레그램 알림 (HIGH)
    "수동 매수 감지: [종목명] [수량]주 @ [가격]"
    "시스템에 추적 등록되지 않은 종목입니다."
    "필요 시 추적 등록을 진행하세요."

MANUAL 포지션 관리:
  시스템 자동 청산 안 함
  사용자가 HTS에서 직접 매도
  매도 감지 시 MANUAL 포지션 수량 감소
  전량 매도 시 status: CLOSED
```

### 7.6 수동 거래 처리 공통 규칙

```yaml
원칙:
  - MANUAL 포지션은 시스템 자동 청산 안 함
  - SYSTEM 포지션은 시스템이 룰대로 관리
  - 두 종류 명확히 분리 (NFR3)

MANUAL 포지션 식별:
  positions.source 필드:
    SYSTEM_A: 경로 A 자동 매수
    SYSTEM_B: 경로 B 자동 매수
    MANUAL: 사용자 수동 매수

주문 태깅:
  키움 계좌는 통합되므로
  주문 발송 시 client_order_id (또는 메모) 활용
  position_id를 태그로 포함
  
  ※ 키움 API 지원 여부 확인 필요
  ※ 미지원 시 대안:
    - 주문 시각 + 수량으로 역매핑
    - 체결 순서 기반 매핑

알림 통합:
  모든 수동 거래는 텔레그램 + 웹 알림
  CRITICAL/HIGH는 웹 알림 아이콘에도 표시
  일일 마감 알림에 수동 거래 내역 포함
```

## §8. WebSocket 끊김 처리

시스템의 생명선인 WebSocket이 끊겼을 때 안전한 복구 절차입니다.

### 8.1 끊김 감지

```yaml
감지 메커니즘:
  - WebSocket 라이브러리의 close 이벤트
  - Heartbeat 타임아웃 (30초 무응답)
  - PING 메시지 응답 없음

감지 즉시:
  - 내부 상태: WEBSOCKET_DISCONNECTED
  - 재연결 시도 시작
  - REST API 폴링 보조 시작
```

### 8.2 재연결 전략

```yaml
Phase 1: 빠른 재연결 (즉시)
  방식: 지수 백오프
  시도 횟수: 5회
  간격: 1s, 2s, 4s, 8s, 16s
  최대 소요: 약 31초

Phase 2: 느린 재연결 (Phase 1 실패 시)
  간격: 5분 고정
  횟수: 무한 (성공할 때까지)
  목적: 키움 서버 점검 시간 등 장기 끊김 대응

재연결 성공 시:
  - 모든 추적 종목 시세 재구독
  - 모든 포지션 REST API로 즉시 재평가
  - 손절/익절 조건 충족 시 즉시 시장가 매도
  - 박스 진입 조건 충족 시 트리거 (단, 지나간 트리거 무효 - §13.1.5)
```

### 8.3 REST API 폴링 (보조 수단)

```yaml
활성화 시점:
  WebSocket 끊김 즉시

대상:
  - 보유 포지션 (POSITION_OPEN, POSITION_PARTIAL)
  - 박스 ±5% 근접 종목 (BOX_SET, NFR1 보장)

주기:
  5초 간격

조회 내용:
  - 현재가 (체결가)
  - 매수/매도 호가 (1~3호가)

비활성화 시점:
  WebSocket 재연결 성공 시 즉시 중지

부하 고려:
  키움 API Rate Limit (초당 4.5회) 준수
  보유/근접 종목만 폴링 (전체 추적 종목 X)
```

### 8.4 긴급정지 없음

```yaml
원칙: 시스템 계속 운영 (헌법 4)

금지:
  - WebSocket 끊김 시 자동 정지
  - 신규 매수 차단 (재연결 후 정상 처리)
  - 시스템 종료

이유:
  - 사용자가 자리에 없을 수 있음
  - 자동 복구가 안전
  - 끊김은 일시적인 경우 대부분
```

### 8.5 알림 정책

```yaml
끊김 10초 이상:
  텔레그램 (MEDIUM)
  "WebSocket 연결 끊김, 재연결 시도 중"

끊김 30초 이상:
  텔레그램 (CRITICAL)
  "WebSocket 30초 이상 끊김, REST API 폴링 활성"

재연결 성공:
  텔레그램 (MEDIUM)
  "WebSocket 재연결 성공"
  소요 시간 표시

재연결 후 발견 사항:
  - 손절 조건 충족 → CRITICAL 알림 + 매도 실행
  - 박스 진입 누락 → CRITICAL 알림 (사후 통보)
```

### 8.6 키움 서버 점검 시간

```yaml
알려진 점검 시간:
  - 매일 새벽 (00:00~06:00 사이 일부)
  - 매주 토요일 새벽
  - 분기별 정기 점검 (사전 공지)

처리:
  - 점검 시간대는 거래 발생 안 함 (장 마감)
  - WebSocket 끊김 자연 발생
  - Phase 2 재연결 (5분 간격)로 점검 종료 시 자동 복구
  - 알림 빈도 제한으로 스팸 방지

특별 처리 불필요:
  점검 시간을 별도 코드로 인식할 필요 없음
  단순 끊김으로 처리
  일반 룰로 처리
```

## §9. 알림 시스템 (Notification System)

사용자가 시스템 상태를 인지하는 핵심 채널입니다.

### 9.1 알림 등급

```yaml
CRITICAL (즉시 확인 필수):
  - 손절 실행
  - 시스템 오류 (거래 정지급)
  - WebSocket 30초+ 끊김
  - 새 IP 로그인 (보안)
  - 자동 이탈 (-20% 이탈)
  - 시스템 재시작 (CRITICAL 상황)

HIGH (중요, 빠른 확인):
  - 매수 실행
  - 익절 실행 (부분/전체)
  - 추매 실행
  - VI 발동
  - 매수 실패
  - 수동 거래 감지

MEDIUM (정보성):
  - 박스 진입 임박 (±5% 이내)
  - WebSocket 10초 끊김 (자동 복구 진행)
  - 박스 만료 임박 (30일 도달)
  - VI 해제
  - WebSocket 재연결 성공

LOW (참고용):
  - 일일 마감 요약 (15:30)
  - 월 1회 추적 리뷰
  - 시스템 헬스 체크
```

### 9.2 알림 채널

```yaml
주 채널: 텔레그램
  단일 채팅방
  authorized_chat_ids에 등록된 ID만 응답

부 채널: 웹 알림 아이콘
  CRITICAL/HIGH 알림은 웹에도 동시 표시
  대시보드 상단 아이콘
  클릭 시 상세 내역

알림 옵션 (사용자 설정):
  CRITICAL: 강제 ON (변경 불가)
  HIGH: ON/OFF 가능
  MEDIUM: ON/OFF 가능
  LOW: ON/OFF 가능
```

### 9.3 우선순위 큐

```yaml
구조: Priority Queue
  CRITICAL > HIGH > MEDIUM > LOW
  같은 등급 내: 시간순 (FIFO)

발송 정책:
  텔레그램 API 제한 준수
    - 초당 1메시지
    - 분당 20메시지
  
  큐에서 우선순위 높은 것부터 pop
  CRITICAL 발생 시 큐 맨 앞으로
  
CRITICAL 강화:
  - 큐 최상위 우선
  - 웹 동시 표시 (큐 무관)
  - Circuit Breaker 차단 시에도 영구 보관
  - 발송 실패 시 5초 후 3회 재시도
  - 모든 재시도 실패 시 ERROR 로그
```

### 9.4 Circuit Breaker

```yaml
목적: 텔레그램 API 장애 시 시스템 자원 보호

동작:
  연속 실패 3회 → Circuit Open (호출 중단)
  대기 30초 → Half-Open (테스트 호출)
  성공 시 → Closed (정상 복귀)
  실패 시 → Open 유지

차단 시 처리:
  CRITICAL/HIGH: 큐에 영구 보관
  MEDIUM/LOW: 5분 내 발송 못 하면 폐기
```

### 9.5 빈도 제한

```yaml
원칙:
  같은 (이벤트 타입 + 종목) 5분 내 1회만 발송

중복 발생 시:
  메시지 압축
  예: "매수 실패 3건 발생, 마지막 14:35"

예외:
  CRITICAL은 빈도 제한 무시 (모두 발송)
```

### 9.6 표준 메시지 포맷

```yaml
구조:
  [등급] [이벤트명]
  필수 정보 (종목, 가격, 시각)
  추가 정보

이모지: 최소화
  텍스트 우선
  호환성 문제 회피

예시 (손절 실행):
  [CRITICAL] 손절 실행
  종목: 에프알텍 (036040)
  매도가: 17,200원 (평단가 18,100원)
  시각: 14:23:45
  
  손익: -4.97% (-2,250,000원)
  사유: 평단가 -5% 도달
  
  남은 박스: 2차 박스 (16,500~17,000원) 무효화
  다음 매수: 추적 종료

예시 (매수 실행):
  [HIGH] 매수 실행
  종목: 주성엔지니어링 (036930)
  매수가: 32,000원 × 50주 = 1,600,000원
  시각: 10:15:32

  박스: 1차 (31,500~32,500원, 비중 10%)
  손절선: 30,400원 (-5%)
```

### 9.7 일일 마감 알림 (15:30)

## §10. VI 처리 (Volatility Interruption)

한국 시장 특유의 변동성 완화 장치 처리입니다.

### 10.1 VI 개요

```yaml
VI (Volatility Interruption):
  급격한 가격 변동 시 일시 정지 장치
  ±10% 변동 시 발동
  2분간 단일가 매매
  이후 정상 거래 재개

종류:
  동적 VI: 직전 체결가 대비 (수시)
  정적 VI: 전일 종가 대비 (시초가 등)
```

### 10.2 VI 감지

```yaml
키움 WebSocket 구독:
  type: '1h'
  대상: 추적 중 모든 종목 (또는 전체 종목)

수신 필드:
  9001: 종목코드
  9068: VI 발동구분 (1: 발동, 2: 해제)
  1221: VI 발동가격
  1224: VI 해제 시각

감지 즉시:
  vi_state 변경: NORMAL → VI_TRIGGERED
  로깅
  텔레그램 알림 (HIGH)
```

### 10.3 VI 상태 머신

```yaml
NORMAL:
  정상 거래 상태
  VI 감지 시 → VI_TRIGGERED

VI_TRIGGERED:
  VI 발동 중 (단일가 매매)
  처리:
    - 보유 포지션: 손절/익절 판정 중단
    - 신규 매수: 단일가 매매 참여 시도
    - 재개 시각 기록

VI_RESUMED:
  VI 해제 직후
  처리:
    - 보유 포지션: 즉시 재평가
    - 손절 조건 충족 시 즉시 시장가 매도
    - 갭 측정 (3% 기준)
  
  → NORMAL (당일 신규 진입 금지 플래그 설정)
```

### 10.4 VI 중 신규 매수

```yaml
상황:
  박스 진입 트리거 충족
  VI 발동 중

처리:
  Step 1: 단일가 매매 참여
    지정가 (이론 매수가)로 주문
    VI 가격 근처에 주문
  
  Step 2: 단일가 체결 결과
    체결: 정상 매수 완료
    미체결: VI 해제 후 재시도
  
  Step 3: VI 해제 후 재시도 (미체결 시)
    갭 측정: (현재가 - VI 발동가) / VI 발동가
    
    갭 < 3%: 시장가 매수
    갭 >= 3%: 매수 포기 + 알림
    
  알림:
    매수 성공: HIGH
    매수 포기: HIGH
      "[종목명] VI 후 갭 X.X%, 매수 포기"
```

### 10.5 VI 중 보유 포지션 처리

```yaml
VI_TRIGGERED 상태:
  손절/익절 판정 중단
  매 틱 판정 일시 정지
  이유:
    - 단일가 매매 중 체결 가능성 낮음
    - 가격 정보 일시적 무의미
    - 잘못된 신호로 매도 발생 위험

VI_RESUMED 상태 (해제 직후):
  즉시 재평가:
    Step 1: 현재가 조회
    Step 2: 손절/익절/TS 조건 체크
    Step 3: 조건 충족 시 즉시 시장가 매도
  
  지연 < 1초 (NFR1)

예외:
  없음 (블랙스완 상황도 손절 룰 그대로)
```

### 10.6 VI 후 신규 진입 금지 (당일)

```yaml
원칙:
  VI 발동 후 당일 신규 진입 금지

적용:
  vi_state가 VI_RESUMED → NORMAL 전환 시
  플래그 설정: vi_recovered_today = True

금지 대상:
  해당 종목의 신규 박스 진입
  (보유 포지션 관리는 정상 진행)

해제 시점:
  익일 09:00 시초가
  플래그 리셋

이유:
  - VI는 시장 이상 신호
  - 당일 다시 들어가는 것 위험
  - 다음날 시장 정상화 대기
```

### 10.7 VI 포함 봉 판정

```yaml
사용자 결정: 봉 데이터 그대로 판정

예시:
  14:30~14:33 3분봉
  14:31 VI 발동 (2분 정지)
  14:33 VI 해제
  14:33 봉 종료
  
  봉 데이터:
    Open, High, Low, Close 그대로 사용
    Close > Open이면 양봉으로 인정

근거:
  - 시스템 단순성 우선 (헌법 5)
  - VI 봉 무효화는 복잡 처리
  - 봉 자체는 형성됨 (체결 적어도 시가/종가 있음)

엣지 케이스:
  Close == Open: 양봉 아님 (조건 미충족, 자연 처리)
  체결 없음: 시가=종가, 양봉 아님
  
  → 별도 처리 불필요, 자연스럽게 조건 미충족
```

### 10.8 상승 VI vs 하락 VI

```yaml
상승 VI (+10%):
  보유 포지션: 수익 증가 (TS 상향 가능)
  신규 진입: VI 후 갭 3% 이상 시 포기
  당일 재진입 금지: 적용

하락 VI (-10%):
  보유 포지션: 손절 조건 충족 가능성 높음
  신규 진입: 일반 룰 (트리거 충족 시 매수)
  당일 재진입 금지: 적용

공통 처리:
  같은 룰 적용
  방향에 따른 차별 없음
```

### 10.9 시초 VI 시나리오 (경로 B 안전장치)

```yaml
배경:
  경로 B는 익일 09:01 매수 (§3.10/§3.11)
  시초 동시호가 결과 ±10% 변동 시 시초가 VI 발동 가능
  시초가 VI는 일반 VI(2분 단일가)와 같은 처리지만,
  09:01 매수 시도가 단일가 영역에 정확히 끼는 빈도가 높음
  → 미체결 종료 위험

해결 (안전장치):
  09:01 1차 매수 시도 (지정가 1호가 위 × 5초 × 3회 → 시장가)
  
  1차 실패 시 (09:04:59까지 미체결):
    09:05 시점에 시장가 재시도
    
    재시도 전 갭업 재검증:
      gap = (09:05 현재가 - 전일 종가) / 전일 종가
      gap < 5%: 시장가 매수
      gap >= 5%: 매수 포기 (안전장치 무력화)
  
  적용 대상:
    - 시초가 VI로 단일가 미체결
    - 09:01 시점 VI 발동 중 (단일가 영역)
    - 키움 API 일시 장애로 09:01 주문 실패
  
  미적용 대상 (1차에서 영구 포기):
    - 갭업 5% 초과 (§3.10 룰)
    - 추적 종료 (EXITED)
    - 거래 정지/상장폐지 위험
    - 한도 초과

VI 후 갭 3% (일반 VI 룰)와 차이:
  일반 VI 후 갭: 3% (§10.4)
  시초 VI 후 갭 (09:05 시점): 5% (PATH_B 갭업 룰과 동일)
  
  근거:
    - 일반 VI는 장중 변동 (이미 정상 가격대 형성됨)
    - 시초 VI는 시초가 자체가 변동 (PATH_B 갭업 룰과 같은 기준 적용 = 일관성)

알림:
  CRITICAL 아닌 HIGH (의도적 안전장치 동작)
  - 안전장치 매수 성공: HIGH
  - 안전장치 시점 갭업 초과 포기: HIGH

VI 당일 재진입 금지 플래그와의 관계:
  09:01 1차 시도가 VI로 미체결 → vi_recovered_today 플래그는 일반 VI 룰대로 설정
  단, 09:05 안전장치는 "이미 트리거된 매수의 마무리"로 보고 진입 금지 플래그 무시
  
  근거:
    - 안전장치는 신규 진입이 아니라 이미 결정된 진입의 재시도
    - 사용자 판단 (전일 일봉 조건 충족) 존중 (헌법 1)
```

## §11. 상하한가 처리

한국 시장의 ±30% 가격 제한 처리입니다.

### 11.1 상한가 임박 매수

```yaml
사용자 결정: 시스템 룰 그대로 매수 (옵션 A)

근거:
  상따 매매도 명확한 전략
  손절 룰 자동 적용 (-5%)
  시스템 단순성 유지

실행:
  현재가 +28% (상한가 직전)
  박스 진입 트리거 충족
  → 정상 매수 실행 (지정가 → 시장가)

위험 인지:
  손익비 불리 가능 (이익 +2%, 손실 -7%)
  사용자가 박스 설정 시 인지하고 입력함
  시스템은 룰만 충실히 실행

대안 옵션 (참고, 미채택):
  옵션 B: 상한가 -5% 이내 매수 차단 (보수)
  사용자가 단순성 우선으로 옵션 A 선택
```

### 11.2 상한가 도달 후 보유 포지션

```yaml
상황:
  포지션 보유 중
  가격 +30% 상한가 도달
  단일가 매매 시작

처리:
  일반 룰 적용 (특별 처리 없음)
  
  분할 익절:
    +5% 도달: 30% 청산 (지정가 → 시장가)
    +10% 도달: 30% 청산
    상한가 단일가에서도 매수 잔량 풍부 → 체결 용이
  
  TS:
    상한가 유지 시 BasePrice 갱신
    하락 시 청산선 도달 가능
  
  손절:
    상한가에서 손절 안 할 가능성 높음 (가격 안 떨어짐)
    익일 갭다운 시 일반 룰 적용

익일 갭다운:
  예: 상한가 마감 후 익일 -10% 갭다운
  
  처리:
    시초가 시점에 손절 조건 즉시 체크
    조건 충족 시 시장가 매도
```

### 11.3 하한가 임박 손절

```yaml
상황:
  매수가 18,000원
  현재가 12,800원 (-29%, 하한가 직전)
  손절 트리거 발동 (이미 -5% 훨씬 넘음)

처리:
  Step 1: 시장가 매도 주문 발송
  Step 2: 매수 잔량 확인
    - 풍부: 정상 체결
    - 부족: 부분 체결 또는 미체결
  Step 3: 미체결분 처리
    주문 유지 (자동 취소 안 함)
    다음 영업일 체결 가능

알림:
  CRITICAL
  "손절 주문 미체결, 하한가/거래정지 상태"
  "수량: X주 미체결"
  "다음 영업일 체결 시도"

블랙스완 손실 감수:
  실제 사례 (전쟁, 횡령, 코로나 등):
    -30% 갭하락 가능
    시스템이 막을 수 없음
    원칙대로 손절 시도, 손실 감수
```

### 11.4 하한가 도달 후 추적

```yaml
자동 이탈 룰로 자연 처리:
  박스 -20% 이탈 자동 EXITED (§2.3)
  하한가 -30%는 이미 -20% 초과
  → 자동 이탈 자동 발동

별도 룰 불필요:
  상태 머신의 자동 이탈이 커버
  하한가만의 특별 룰 없음

포지션 처리:
  손절 시도 (시장가)
  미체결 시 대기
  자동 이탈은 추적만 종료, 강제 청산 안 함
```

### 11.5 거래 정지 처리

```yaml
상황:
  악재로 거래 정지
  매도 시도 안 됨

처리:
  자동 이탈 조건 충족 (거래 정지)
  tracked_stocks.status: EXITED
  
  포지션:
    매도 주문 발송됐어도 미체결
    주문 유지
    거래 재개 시 시장가 매도 시도
  
알림:
  CRITICAL
  "거래 정지 감지, 자동 이탈"
  "포지션 N주 보유 중, 거래 재개 시 청산 시도"

거래 재개:
  키움 API 상태 변화 감지
  미체결 주문 자동 시장가 매도
  체결 결과 알림
```

## §12. 주문 취소 및 재시도

주문 처리의 모든 엣지 케이스입니다.

### 12.1 시장가 주문 실패

```yaml
상황:
  지정가 3회 시도 후 시장가 전환
  시장가도 미체결 (드물지만 가능)

원인:
  - 키움 API 일시 장애
  - 거래 정지 직전
  - 호가 자체가 비어있을 때
  - 잔고 부족

처리:
  Step 1: 5초 대기 후 시장가 1회 재시도
  Step 2: 또 실패 시 텔레그램 알림 (HIGH)
    "[종목명] 시장가 매수 실패, 사유: [에러]"
    "수동 확인 필요"
  Step 3: 자동 재시도 중단
    무한 루프 방지
  Step 4: 사용자 수동 개입 대기

박스 처리:
  박스 유지 (다음 트리거 대기)
  잔고 부족 등 영구 사유는 사용자 결정 필요
```

### 12.2 부분 체결 처리

```yaml
상황:
  지정가 100주 주문
  50주 체결, 50주 미체결
  대기 시간 5초 종료

처리:
  Step 1: 미체결 50주 취소
  Step 2: 시도 카운트 +1
  Step 3: 미체결 수량으로 재시도 사이클 계속
    시도 1 → 시도 2 (50주 새 지정가)
  Step 4: 3회 모두 미체결 시 시장가 전환

평단가 처리:
  부분 체결분도 즉시 평단가 반영
  
  예시:
    50주 × 18,000 = 900,000
    이후 50주 × 18,050 = 902,500
    합계: 100주, 평단가 18,025
```

### 12.3 가격 급변 시 처리

```yaml
상황:
  지정가 18,000원 매수 주문 발송
  2초 후 가격 17,500원 급락
  지정가가 너무 비쌈

처리:
  사용자 결정: 옵션 A (그대로 대기)
  
  실행:
    5초 대기 끝까지 유지
    가격 반등 시 체결, 안 되면 시장가
    3회 시도 + 시장가로 결국 체결

근거:
  - 단순함 우선 (헌법 5)
  - 가격 추적 갱신은 복잡
  - 시장가가 결국 실행됨
```

### 12.4 API 응답 타임아웃

```yaml
상황:
  주문 요청 발송
  10초 이상 응답 없음
  실제로 주문이 들어갔는지 모름

처리:
  Step 1: 미체결 주문 조회 (키움 API)
  Step 2: 해당 주문 존재 확인
    존재: 정상 처리 (체결 대기)
    부재: 재요청
  Step 3: 5회 재시도 후도 실패 시
    텔레그램 알림 (CRITICAL)
    사용자 수동 개입 요청

안전 장치:
  타임아웃 시간: 10초
  재시도 간격: 1초
  최대 시도: 5회
  총 소요: 약 50초
```

### 12.5 박스 삭제 시 미체결 주문

```yaml
상황:
  지정가 매수 주문 발송 (대기 중)
  사용자가 웹에서 해당 박스 삭제

처리:
  사용자 결정: 옵션 A (자동 취소)
  
  실행:
    Step 1: 박스 삭제 요청 수신
    Step 2: 해당 박스의 미체결 주문 조회
    Step 3: 모두 취소
    Step 4: 박스 status: CANCELLED
    Step 5: 알림 (MEDIUM)
      "박스 삭제, 미체결 주문 N건 취소"

근거:
  박스 삭제 = 매수 의도 철회
  주문 유지 시 의도와 모순
  자동 취소가 일관됨
```

### 12.6 주문 취소/재시도 일관 룰

```yaml
원칙:
  - 모든 주문은 추적 가능
  - 주문 ID와 position_id 매핑 유지
  - 모든 시도/재시도 로깅

로깅 항목:
  trade_events:
    event_type: ORDER_SENT, ORDER_FILLED, ORDER_CANCELLED, ORDER_FAILED
    order_id, attempt, price, quantity, timestamp, error

  주문 추적 시 키움 API의 client_order_id 활용
  (지원 여부 확인 필요)
```

## §13. 시스템 재시작 복구

시스템이 다양한 이유로 재시작될 때 안전한 복구 절차입니다.

### 13.1 복구 시퀀스

```yaml
Step 0: 안전 모드 진입
  부팅 즉시 실행
  처리:
    - 신규 매수 주문 차단
    - 신규 박스 등록 차단
    - 추적 모니터링 시작 (시세 구독은 가능)
  알림: "시스템 복구 중, 약 30초~2분 소요"

Step 1: 외부 시스템 연결
  순서:
    1. DB 연결 (Supabase)
    2. 키움 OAuth 토큰 검증/갱신
    3. WebSocket 연결
    4. 텔레그램 봇 연결
  
  실패 시:
    각 5회 재시도
    모두 실패 시 알림 + 정지

Step 2: 미완료 주문 처리
  사용자 결정: 옵션 A (모두 취소)
  
  실행:
    1. 키움 API에 떠 있는 미체결 주문 조회
    2. 모두 취소
    3. 박스 설정은 영향 없음 (DB 보존)
  
  중요:
    박스는 그대로 유지
    임시 주문만 정리
    다음 트리거 시 새 주문 발송

Step 3: 포지션 정합성 확인 (Reconciliation)
  키움 ↔ DB 대조
  
  Case A: 일치
    그대로 진행
  
  Case B: 키움 > DB (DB에 없는 보유)
    MANUAL 포지션 신규 등록
    텔레그램 알림
  
  Case C: 키움 < DB (DB에 있는데 키움에 없음)
    시스템 포지션 수량 감소
    평단가 유지
    MANUAL_PARTIAL_EXIT 이벤트 기록
  
  Case D: 키움 0, DB 있음
    EXITED 처리
  
  Case E: 키움 있음, DB 0
    MANUAL 포지션 신규
  
  이중 경로 차이:
    자동 비례 차감
  
  모든 차이: 텔레그램 알림 (HIGH)

Step 4: 시세 재구독
  대상:
    - BOX_SET 종목 (박스 근접도에 따라)
    - POSITION_OPEN, POSITION_PARTIAL 종목 (모두)
  
  WebSocket 재연결 후 일괄 재구독

Step 5: 박스 진입 조건 재평가
  사용자 결정: 옵션 A (지나간 트리거 무효)
  
  로직:
    재시작 동안 박스 진입 발생 가능
    그러나 트리거 시점 이미 지남
    → 매수 실행 안 함
    → 다음 정상 트리거 대기
  
  현재 시점 재평가:
    현재 봉이 박스 내 + 양봉 조건이면
    다음 봉 완성 시점에 정상 처리
  
  근거:
    단순함 우선
    손실 위험 회피
    박스 살아있으니 다음 기회

Step 6: 안전 모드 해제
  - 신규 매수 차단 해제
  - 박스 등록 차단 해제
  - 정상 운영 시작

Step 7: 복구 보고서
  텔레그램 알림 (CRITICAL)
  
  내용:
    [시스템 복구 완료]
    복구 시작: HH:MM:SS
    복구 완료: HH:MM:SS (XX초 소요)
    
    처리 내역:
      - 미완료 주문 취소: N건
      - 키움 ↔ DB 대조: 일치 / 차이 N건 처리
      - WebSocket 재구독: M개 종목
      - 박스 조건 재평가: 변동 없음
    
    정상 운영 시작
```

### 13.2 재시작 빈도 모니터링

```yaml
사용자 결정: 모니터링 적용, 자동 정지 없음

구현:
  최근 재시작 이력 DB 기록
  system_restarts 테이블
  
  필드:
    - id, restart_at, recovery_duration
    - reason (KNOWN_DEPLOY / CRASH / MANUAL / ETC)
    - reconciliation_summary

빈도별 알림:
  1회 재시작: 일반 처리
  2회 (1시간 내): 경고 알림
    "재시작 빈도 비정상 가능성, 모니터링 강화"
  
  3회 (1시간 내): CRITICAL 알림
    "재시작 빈도 비정상, 시스템 불안정 가능성"
    "사용자 인지 후 수동 점검 권장"
  
  5회 이상 (1시간 내):
    ERROR 로그 기록
    CRITICAL 알림 반복
    하지만 시스템 계속 가동

자동 정지 안 함 (헌법 4):
  텔레그램 폭주로 사용자 자연 인지
  본인 판단으로 수동 점검
```

### 13.3 복구 실패 케이스

```yaml
키움 API 연결 실패:
  5회 재시도
  실패 시 알림 + 정지
  사용자 수동 개입 필요

DB 연결 실패:
  5회 재시도
  실패 시 알림 + 정지

대조 결과 비정상:
  예: 시스템에 모르는 포지션 5개 키움에 있음
  → 모두 MANUAL로 등록
  → 사용자 검토 알림
  → 정상 운영 시작 (정지 안 함)

WebSocket 연결 실패:
  REST API 폴링으로 임시 대응
  Phase 2 무한 재시도 진행
  정상 운영 (제한적)
```
